01

Where momentum dies in product work

The design → code handoff is not a people problem — it's a tooling problem. Every step forces manual re-interpretation, and each gap is where time, intent, and design system compliance are lost.

📐

Designers spec. Developers guess.

Even with Figma, intent gets lost. Component names differ from code. Token values drift. Each interpretation adds a review cycle.

🧩

Design system exists. Nobody uses it in code.

We built the DS. But without a forcing function at code-generation time, developers write their own CSS instead of pulling from tokens.

⏱

Review cycles eat sprints.

3–5 back-and-forth rounds per screen is normal. Each round is async, slow, and introduces new drift between design and implementation.

♿

Accessibility caught too late.

Contrast issues, missing ARIA labels, and semantic errors are found in QA — or worse, by users. By then they're expensive to fix.

04

What Razorpay's Blade taught us

Razorpay built Blade — a fully public, open-source design system — to standardize UI across their product ecosystem. It's the clearest example of what happens when a design system is treated as first-class infrastructure, not an afterthought.

blade.razorpay.com — why it's the reference

Blade is Razorpay's open-source design system: 60+ components, live Storybook docs, Figma libraries, design tokens, and contribution guidelines — all public. It enforces consistency across 200+ engineers without creating a design bottleneck. Every component decision is documented, versioned, and accessible by default. The result: zero design-developer translation loss at scale.

Razorpay · Blade
  • Web + React Native UI components
  • Open-source, external contributors
  • UI toolkit is the core product
  • Designers control the component API
  • Storybook as documentation layer
  • External developer ecosystem
Skillz · Our opportunity
  • SDK-first: we abstract the UI layer
  • DS value is internal team velocity
  • AI enforces DS at generation time
  • Same philosophy, right form factor
  • Our docs site is our "Blade equivalent"
  • Game devs own their art direction
The key distinction — why we don't build Blade directly

Skillz game developers integrate via SDK, so a React component library would be redundant — they own their own art direction and game UI. Our design system value is entirely internal: it makes our own team faster and more consistent. What Blade does for Razorpay's web product, our DS + AI workflow does for our internal product team. Same outcome, right-sized implementation.

Why catching this at design time matters

A contrast issue found in QA requires a designer fix, a developer change, a new PR, a new deploy, and another QA check. Found at the design stage: one token swap, done in 10 seconds.

The next layer — what becomes possible

What's live today is the foundation. Once this workflow is standardized across the team, these are the natural next capabilities — each one compounding on the last.

🧠

CLAUDE.md in repo

Persistent AI memory of our design system. Every Claude Code session auto-knows our tokens, components, and rules — no re-prompting each time.

📋

Prompt library

Standardized prompts for common screen types — stored in the repo, versioned, improvable by anyone. Democratizes the workflow beyond the person who set it up.

🔍

DS compliance in PRs

Claude flags token drift and DS violations in code review automatically. Enforcement moves from "please follow the DS" to "the PR won't pass if you don't."

♿

Accessibility at spec time

Contrast ratios, ARIA roles, semantic structure — checked at the design stage. Issues cost nothing to fix when they're caught before code is written.

📝

Auto-docs generation

When a new component is built in code, Claude drafts the documentation page. DS docs stay current without a dedicated documentation sprint.

🔄

Design ↔ code feedback loop

When code diverges from design, Claude diffs intent vs implementation and proposes token fixes — closing the loop instead of letting drift accumulate.

The ask

Not asking for budget or new hires. Asking for mandate — permission to formalize and scale what already works.

Four things to make this the team standard

1
Formalize this as the team default — not optional, not just the designer who set it up. It becomes how we work.
2
Invest 1 sprint to build the CLAUDE.md and prompt library so the whole team benefits from the foundation already built.
3
Onboard 2–3 more designers onto Claude Code + Figma MCP with structured training — not ad hoc self-discovery.
4
Define "AI-assisted design review" as a step in the sprint process — scheduled, documented, and consistently applied.
What success looks like in 60 days

Every designer on the team can go from Figma frame to Vercel preview without writing a spec doc. DS compliance rate measurably up. Review cycles measurably down.

What failure looks like

We keep this as one designer's personal workflow. The tooling exists, the opportunity exists, but the team gets none of the compounding benefit because it was never standardized.
